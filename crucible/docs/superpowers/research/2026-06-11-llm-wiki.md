# llm-wiki concepts: wiki-structured knowledge substrates for LLM agents (research notes)

Date: 2026-06-11. In-repo notes, not published. Survey of agent-maintained
wikis as memory (Dec 2025 - Jun 2026, plus founding work), comparison with
vector stores and knowledge graphs, editing disciplines, retrieval, and what
any of it means for crucible.

## Summary

An "LLM wiki" is a directory of markdown pages that an agent itself writes and
maintains: one page per entity or concept, wikilinks between pages, an index
file as the catalog, an append-only log, immutable raw sources underneath, and
a schema document that encodes the editing policy. Karpathy's April 2026 gist
crystallized the pattern and a small ecosystem (repos, skills, tutorials) grew
around it within weeks. The core bet, versus RAG, is compile-once: ingest does
the synthesis work and queries read curated pages, instead of re-deriving
answers from raw chunks every time. Evidence that plain files plus an agent
that can grep is competitive predates the gist (Letta's LoCoMo benchmark).
Page granularity wins for small-to-medium corpora and synthesis-heavy queries;
entity/relation graphs (GraphRAG, HippoRAG) still win for large-corpus
multi-hop retrieval, and the hosted-memory vendors (Mem0, Zep) ship
vector+graph hybrids and ignore wikis entirely. The gist's own comment thread
(887 comments by June 10; Karpathy silent throughout, most comments LLM-written
promotion, roughly 35-40 substantive) converged from practice on two gaps:
epistemic integrity (citation-first, review-gated writes, hash-checked
provenance) and the few-hundred-page index ceiling, and produced one genuinely
new design idea, typed contradiction edges paired with a
preserve-the-disagreement lint policy.

## Concepts

### 1. The agent-maintained wiki pattern

**Karpathy's llm-wiki gist (April 2026)** is the reference design
([gist](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)).
Three layers:

- **Raw sources**, immutable. The agent reads them, never edits them. They are
  the ground truth that pages cite back to.
- **The wiki**, LLM-owned: markdown pages for sources (one summary page per
  source document), entities (people, companies, projects), concepts, and
  syntheses (good query answers filed back as pages, so answering compounds
  the knowledge base).
- **The schema** (a CLAUDE.md-style document) that defines structure
  conventions, the ingest workflow, maintenance rules, and contradiction
  handling. Karpathy calls the schema load-bearing: it, not code, determines
  the editing discipline.

Bookkeeping artifacts: `index.md`, a content catalog with one line per page,
updated on every ingest; `log.md`, an append-only chronological record of
ingests, queries, and lint passes. Revision history and provenance come from
git, which the wiki lives in; the design deliberately reuses git rather than
inventing version tracking. Three operations: **ingest** (read source, update
the 10-15 pages it touches, update index, append log), **query** (read index,
load relevant pages, synthesize with citations, optionally file the answer
back), **lint** (periodic health pass for contradictions, stale claims, orphan
pages, missing cross-references).

### The gist's comment thread

The gist accumulated 887 comments between April 4 and June 10, 2026
([API](https://api.github.com/gists/442a6bf555914893e9891c11519de94f/comments)).
Karpathy never replied, so all evolution is commenter-driven. Read it with a
filter: [joshwand (Apr 12)](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f?permalink_comment_id=6094660#gistcomment-6094660)
estimated over 90 percent of comments are LLM-written, and the bulk is release

carry real conceptual content. What they add:

**Epistemic integrity is the consensus gap.** The canonical statement is
[laphilosophia (Apr 4)](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f?permalink_comment_id=6079377#gistcomment-6079377):
the risky sentence in the gist is "the LLM owns this layer entirely," and the
robust version is a "source-grounded, citation-first, review-gated wiki" where
the LLM proposes diffs rather than owning truth; separate facts, inferences, and
open questions; make ingest idempotent. Mechanisms followed.
[Jwcjwc12 (Apr 5)](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f?permalink_comment_id=6082231#gistcomment-6082231)
made provenance structural: every compiled claim records its source files and
their content hashes at compile time, and every read re-checks the hashes, so
staleness is detected mechanically instead of by lint guesswork (git branch
switches then invalidate the right claims for free). Outside evidence got
posted too: [asakin (Apr 12)](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f?permalink_comment_id=6093103#gistcomment-6093103)
cites an ETH Zurich result that LLM-generated context files hurt agent
performance in 5 of 8 tested settings because the model invents its own schema
and tag formats as it goes (their fix: a human-reviewed training period before
autonomy), and [mauceri (Apr 24)](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f?permalink_comment_id=6116491#gistcomment-6116491)
posted "LLMs Corrupt Your Documents When You Delegate"
([arXiv 2604.15597](https://arxiv.org/abs/2604.15597)). The most-adopted
implementation born in the thread, ethanj's
[llm-wiki-compiler](https://github.com/atomicmemory/llm-wiki-compiler)
(1k stars by [May 5](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f?permalink_comment_id=6135386#gistcomment-6135386)),
shipped the review gate as `compile --review` plus claim-level citations of the
form `^[paper.md:42-58]`.

**Production rules for the schema.**
[bluewater8008 (Apr 5)](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f?permalink_comment_id=6079549#gistcomment-6079549),
running the pattern in production for weeks, posted the most-cited rule set:
classify sources by type before extracting; give the index an explicit token
budget (four levels, roughly 200-token project context, 1-2k index, 2-5k search
results, full pages last); one template per entity type rather than one generic
template (seven types was their sweet spot); and "every task produces two
outputs", the answer plus the wiki updates, otherwise knowledge evaporates into
chat history. [xoai (Apr 6)](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f?permalink_comment_id=6082547#gistcomment-6082547)
reached the two-output rule independently while building
[sage-wiki](https://github.com/xoai/sage-wiki) (a single-Go-binary
implementation with compile/search/query/lint and an MCP server) and added:
the compiler wants to be a pipeline of small incremental passes, not one
prompt; concept deduplication ("is attention mechanism the same node as
self-attention?") is the hardest problem; keep unresolved "red link" wikilinks
rather than stripping them.
[benjimixvidz (Apr 15)](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f?permalink_comment_id=6102240#gistcomment-6102240)
attacked growth: append-only is a trap, so every page gets a line budget and an
Actuel/Archive split where superseded content compresses to one dated line
(project wiki capped near 300 lines total), with the stable auto-loaded
CLAUDE.md kept separate from a rewritten state-snapshot page and the on-demand
deep wiki. [KevinYoung-Kw (Apr 16)](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f?permalink_comment_id=6104347#gistcomment-6104347)
added a mechanical update-vs-create rule: same canonical question plus over 60
percent entity overlap means update the existing page, otherwise create one.

**The index ceiling and the split of labor.** Independent reports put the
single-index breaking point at a few hundred pages
([mpazik Apr 5](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f?permalink_comment_id=6079689#gistcomment-6079689),
[skynet Apr 20](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f?permalink_comment_id=6110770#gistcomment-6110770)),
consistent with the v2 gist's number.
[gulliveruk (Apr 20)](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f?permalink_comment_id=6109752#gistcomment-6109752)
explains why: index-guided browsing makes the LLM do scoping, reading, and
reasoning at once, and "scoping should be deterministic, reasoning should be
probabilistic"; the fix is a derived graph with human-defined edge vocabulary
used as a deterministic retrieval filter, markdown staying canonical.
foundanand's "[Hidden Flaw](https://foundanand.medium.com/the-hidden-flaw-in-karpathys-llm-wiki-e3a86a94b459)"
post (Apr 18, linked in thread) gives the memorable form: a librarian who
writes index cards pointing at the originals stays auditable; one who writes
new books and shelves them next to the originals does not.
[yogirk (Apr 22)](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f?permalink_comment_id=6113426#gistcomment-6113426)
split the labor on a different axis: the agent is good at reading and writing
content and bad at plumbing (hashing files, splitting entries, regenerating
indexes, where it burns tokens), so the mechanical layer moved into a small
binary ([sparks](https://github.com/yogirk/sparks)) and the agent instructions
collapsed to about three lines. skynet's same comment lists the other gaps the
gist leaves unstated: large sources need a pre-retrieval step at ingest (RAG as
scaffolding for ingest, not as the query interface), and contradiction
resolution is left entirely to the schema author.

**Contradiction policy got the most genuinely new design thinking.**
[iqsource (Apr 30)](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f?permalink_comment_id=6128209#gistcomment-6128209),
after 18 days running the pattern for a services business, proposed
contradiction policy at the source-citation level: the wiki must refuse to
silently pick a winner when two sources disagree.
[pursultani (Jun 5)](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f?permalink_comment_id=6183984#gistcomment-6183984)
went further: the pattern assumes a convergent epistemology where contradiction
is a defect, which is wrong in dialogic domains (criticism, philosophy,
historiography) where contradiction is the content; there the lint should flag
missing contradictions, not existing ones. The fix has two parts that only work
together: typed edges (a frontmatter `relates_to` block with
`rel: contradicts | extends | supersedes`) and a schema policy that preserves
tension; a `contradicts` edge is worthless if lint still treats it as cleanup.
Bare wikilinks cannot represent any of this, which is the one structural gap in
the original design the thread actually nailed down.

**Multi-writer wikis.**
[watsonrm (Jun 5)](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f?permalink_comment_id=6184491#gistcomment-6184491):
git branch-and-merge solves textual conflicts, but the corruption that wrecks a
shared wiki is semantic, two agents writing the same fact in different words and
merging cleanly into a duplicate. The fix is data modeling, not coordination:
partition writers per file or section, keep writes append-only so they commute,
key dedup and contradiction detection on a stable claim identity (the source
citation) rather than textual proximity, and route genuine conflicts to a
review queue instead of letting an LLM auto-resolve them. Idempotent,
citation-keyed ingest also dissolves the ingestion-order bias that
[brtrx (May 28)](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f?permalink_comment_id=6172613#gistcomment-6172613)
patched with a batched, randomized-order lint
([their lint extension](https://gist.github.com/brtrx/ba595fd01e344d797cb66d34d982ecad)).
mpazik's [Binder](https://github.com/mpazik/binder) is the inversion answer to
the same pressures: structured data in a transaction log rendered as editable
markdown, with the index as a query instead of a maintained file.

**Benchmark skepticism stands.**
[a-a-k (May 2)](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f?permalink_comment_id=6130180#gistcomment-6130180)
is the thread's best critique: no benchmark, no task definition, no scale
curve, no comparison against hybrid RAG or GraphRAG baselines; the wiki is
lossy compression, and once people query it instead of the sources, summary
errors become the knowledge base. The only quantitative data point posted is
[Eyaldavid7's (Apr 10)](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f?permalink_comment_id=6091440#gistcomment-6091440)
informal head-to-head on a 50k LOC codebase
([writeup](https://open.substack.com/pub/eyal454160/p/why-your-ai-agent-needs-a-wiki-and)):
the wiki beat RAG on questions about deleted code (institutional memory RAG
cannot have), a partially compiled wiki scored 17 percent worse than a fully
compiled one (compile quality is binary, not gradual), and a combined
wiki-plus-RAG setup never lost a round. Small and informal, but it matches the
section 2 observation: nobody has published wiki numbers on a standard memory
benchmark yet.

- A "v2" extension proposal

  confidence/decay scores, and hybrid search. Its own comment thread flags the

- Practical comparisons: [MindStudio on wiki vs RAG for codebase memory](https://www.mindstudio.ai/blog/llm-wiki-vs-rag-internal-codebase-memory),
  [Vishal Mysore's RAG vs agent memory vs wiki](https://medium.com/@visrow/rag-vs-agent-memory-vs-llm-wiki-a-practical-comparison-41a9a0dc4dec),
  [Data Science Dojo tutorial](https://datasciencedojo.com/blog/llm-wiki-tutorial/).

**Founding lineage** (the gist names none of these, but they are the priors):
Generative Agents introduced an agent-maintained memory store with periodic
reflection that writes higher-level synthesized records
([arXiv 2304.03442](https://arxiv.org/abs/2304.03442), 2023). MemGPT framed
memory as paging between context and external storage
([arXiv 2310.08560](https://arxiv.org/abs/2310.08560), 2023). A-MEM is the
closest academic ancestor: Zettelkasten-style atomic notes where each new note
gets LLM-generated keywords, tags, links to related notes, and triggers
"memory evolution," updates to existing notes' contents when new information
arrives ([arXiv 2502.12110](https://arxiv.org/abs/2502.12110), Feb 2025;
a good walkthrough at
[Alpha's Manifesto, Apr 2026](https://blog.alphasmanifesto.com/2026/04/11/a-mem-zettelkasten-for-agents/)).
ACE (Stanford/SambaNova/Berkeley, Oct 2025) is the same idea for procedural
knowledge: an evolving "playbook" context maintained by generator, reflector,
and curator roles ([arXiv 2510.04618](https://arxiv.org/abs/2510.04618)).

### 2. Wiki structure vs alternatives

**Vs vector RAG.** RAG retrieves top-k similar chunks at query time; nothing
accumulates, and synthesis across fragments is redone per query. The wiki
moves that work to ingest. The wiki side also avoids embedding infrastructure
below roughly 100k tokens of wiki, where the index plus selected pages fit in
context ([Karpathy gist](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f);
[MindStudio comparison](https://www.mindstudio.ai/blog/llm-wiki-vs-rag-internal-codebase-memory)).
RAG keeps the advantage at enterprise document counts and for corpora that
change faster than an agent can re-compile pages.

**Vs knowledge graphs.** GraphRAG (Microsoft,
[arXiv 2404.16130](https://arxiv.org/abs/2404.16130)) and HippoRAG
([arXiv 2405.14831](https://arxiv.org/abs/2405.14831),
[repo](https://github.com/osu-nlp-group/hipporag)) extract entity/relation
triples and retrieve via community summaries or personalized PageRank. They
win on multi-hop questions over large corpora and HippoRAG 2 indexes much
cheaper than GraphRAG
([graphwise.ai](https://graphwise.ai/blog/from-retrieval-to-reasoning-enhancing-hipporag-with-graph-based-semantics/)).
Two caveats favor pages. First, triple extraction discards prose nuance;
a page holds hedges, context, and contradictions that a triple cannot.
Second, the graph advantage shrinks under agentic retrieval, where the agent
natively does iterative multi-turn querying, which is exactly the wiki's
retrieval mode (observation made in the LinearRAG line of work,
[arXiv 2510.10114](https://arxiv.org/pdf/2510.10114), and visible in
[Letta's benchmark](https://www.letta.com/blog/benchmarking-ai-agent-memory)).

**Files + grep is a strong baseline.** Letta showed a GPT-4o-mini agent with
only file operations (grep, search, open) scoring 74.0% on LoCoMo, beating
the specialized memory vendors' published numbers at the time; their claim is
that what matters is whether the agent can drive the retrieval tool, and file
tools are heavily represented in training data
([Letta, Aug 2025](https://www.letta.com/blog/benchmarking-ai-agent-memory)).

**The hosted-memory view.** Mem0's "State of AI Agent Memory 2026"
(June 2026) reports the production frontier as vector+graph hybrid with
multi-signal fusion (semantic + BM25 + entity match) and does not mention
file- or wiki-based memory at all
([mem0.ai](https://mem0.ai/blog/state-of-ai-agent-memory-2026)). Read that as
a vendor lens, but also as honest signal: nobody has published wiki numbers
on LoCoMo/LongMemEval-class benchmarks yet. Its open-problems list (staleness
of high-relevance facts, modeling change as evolution rather than
replacement) is exactly what the wiki pattern's supersession and lint rules
attack from the other direction.

**When page granularity wins:** corpus small enough that an index fits in
context; queries that need synthesis and narrative, not point lookup; a human
who wants to audit or co-edit the memory (markdown is reviewable, embeddings
are not); provenance requirements (page text cites source pages, git blames
every line). When it loses: very large or fast-churning corpora, point
lookups at scale, multi-tenant memory services.

### 3. Editing disciplines

The schema document carries the policy. Concrete rules from
[llm-wiki-agent's AGENTS.md](https://github.com/SamurAIGPT/llm-wiki-agent/blob/main/AGENTS.md):

- **Create**: a source page per ingested document; an entity page once an
  entity is mentioned by 3 or more sources; a concept page per core idea; a
  synthesis page per answered query worth keeping.
- **Update**: when a new source contradicts or extends prior claims, edit the
  existing page (and record the contradiction) rather than fork a new one.
- **Merge across layers: never.** Source, entity, and concept layers stay
  separate so provenance survives; entity pages cite source pages, never raw
  text directly.
- **Lint**: orphan pages, broken wikilinks, pages with fewer than 2 outbound
  links (flagged sparse), contradictions, stale summaries, missing entity
  pages, plus graph-aware checks (hub stubs, isolated communities).

**Conflict and supersession.** The v2 gist's rule: a new claim that
contradicts an old one should explicitly supersede it, "linked, timestamped,
old version preserved but marked stale," not silently overwrite and not
accumulate as noise
([gist](https://gist.github.com/rohitg00/2067ab416f7bbe447c1977edaaa681e2)).
Mem0's pipeline frames the same decision as a classification per incoming
fact: ADD, UPDATE, DELETE, or NOOP against existing memories
([arXiv 2504.19413](https://arxiv.org/abs/2504.19413)). LangMem does
consolidation automatically, merging related facts and resolving
contradictions ([conceptual guide](https://github.com/langchain-ai/langmem/blob/main/docs/docs/concepts/conceptual_guide.md)).

**Avoiding rewrite damage.** ACE identifies two failure modes of letting a
model rewrite its own knowledge wholesale: brevity bias (rewrites compress
away detail) and context collapse (iterative full rewrites erode content).
Its fix is delta updates: the curator emits small add/update items and never
regenerates the whole document
([arXiv 2510.04618](https://arxiv.org/abs/2510.04618),
[VentureBeat](https://venturebeat.com/ai/ace-prevents-context-collapse-with-evolving-playbooks-for-self-improving-ai)).
For a wiki this argues for section-level edits and append-bias over full-page
regeneration.

**When maintenance runs.** Letta's sleep-time agents do consolidation
asynchronously: a background agent shares the memory of the primary agent and
rewrites it between tasks, which is the wiki's lint/ingest pass as an
architecture ([Letta](https://www.letta.com/blog/sleep-time-compute),
[docs](https://docs.letta.com/guides/agents/architectures/sleeptime/)).

**Citations back to traces.** In llm-wiki-agent, source pages carry
frontmatter with the raw `source_file`, plus Key Claims and Key Quotes
sections; every synthesized answer cites wiki pages as wikilinks, and those
pages cite sources. So the provenance chain is answer -> page -> source page
-> immutable raw file, with git history as the revision record.

### 4. Retrieval over wikis

The default retrieval is **index-guided browsing**: the agent reads
`index.md` (every page title plus a one-line summary), picks pages, reads
them, follows wikilinks as needed. Page titles function as a humanly and
model-legible inverted index; no embeddings until the wiki outgrows context
([Karpathy gist](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)).

This is the same mechanism as **PageIndex** (Vectify), which builds a
table-of-contents tree over long documents and has the model navigate it by
reasoning, section by section, instead of similarity search; it reports
strong FinanceBench results
([pageindex.ai](https://pageindex.ai/blog/pageindex-intro),
[repo](https://github.com/VectifyAI/PageIndex),
[MarkTechPost, Apr 2026](https://www.marktechpost.com/2026/04/25/rag-without-vectors-how-pageindex-retrieves-by-reasoning/)).
Earlier academic versions of structure-navigating retrieval: MemWalker
(interactive tree traversal, [arXiv 2310.05029](https://arxiv.org/abs/2310.05029)),
ReadAgent (gist-page memory with lookup,
[arXiv 2402.09727](https://arxiv.org/abs/2402.09727)), RAPTOR (hierarchical
summary tree, [arXiv 2401.18059](https://arxiv.org/abs/2401.18059)).

Lexical search (grep/BM25) is the second tool and is what made Letta's
filesystem baseline work; the agent iterates queries until it finds the
page ([Letta](https://www.letta.com/blog/benchmarking-ai-agent-memory)).
Semantic search gets bolted on only at scale; v2 proposes BM25 + vectors +
graph traversal fused with reciprocal rank fusion
([gist](https://gist.github.com/rohitg00/2067ab416f7bbe447c1977edaaa681e2)).
The ordering everyone converges on: titles/index first, lexical second,
embeddings last.

### 5. Labs and frameworks treating pages/documents as the unit

- **Anthropic memory tool**: a client-side tool where Claude creates, reads,
  updates, and deletes files under a `/memories` directory across sessions;
  the host app executes the operations and must path-validate. Anthropic
  injects "ALWAYS VIEW YOUR MEMORY DIRECTORY BEFORE DOING ANYTHING ELSE" into
  the system prompt. Beta header `context-management-2025-06-27`
  ([docs](https://platform.claude.com/docs/en/agents-and-tools/tool-use/memory-tool)).
  The unit of memory is a file the model edits, exactly the wiki substrate
  minus the schema discipline.
- **Claude Code memory**: hierarchical CLAUDE.md files (user, project,
  directory) as persistent instruction pages
  ([docs](https://code.claude.com/docs/en/memory)).
- **Anthropic Agent Skills**: folders with a SKILL.md plus resources, loaded
  by three-level progressive disclosure (frontmatter ~100 tokens always in
  context, body on demand, bundled files on demand)
  ([engineering post](https://www.anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills),
  [docs](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/overview)).
  Progressive disclosure is the index-then-page retrieval pattern applied to
  procedural knowledge.
- **Letta**: memory blocks (labeled, size-bounded context sections, shareable
  between agents) for in-context memory
  ([blog](https://www.letta.com/blog/memory-blocks)), and MemFS, a git-backed
  directory of markdown files, as the persistent layer in Letta Code
  ([blog](https://www.letta.com/blog/letta-code),
  [docs](https://docs.letta.com/letta-code/memory/)).
- **LangChain LangMem**: semantic memory as either **collections** (many
  small records) or **profiles** (a single structured document with a schema,
  continuously patched). The profile is a typed page: schema-validated,
  updated in place
  ([conceptual guide](https://github.com/langchain-ai/langmem/blob/main/docs/docs/concepts/conceptual_guide.md),
  [LangChain long-term memory docs](https://docs.langchain.com/oss/python/langchain/long-term-memory)).
- **OpenAI**: no wiki-shaped product. ChatGPT memory is an automatic fact
  store plus a file library ([Memory FAQ](https://help.openai.com/en/articles/8590148-memory-faq),
  [TechRadar on document memory](https://www.techradar.com/ai-platforms-assistants/chatgpt/openai-just-gave-chatgpt-a-memory-for-your-documents-and-free-users-get-it-too));
  the Agents SDK added configurable memory, compaction, and filesystem tools
  ([OpenAI](https://openai.com/index/the-next-evolution-of-the-agents-sdk/),
  [cookbook](https://developers.openai.com/cookbook/examples/agents_sdk/building_reliable_agents_memory_compaction)),
  and the AGENTS.md convention puts standing instructions in workspace files.
  Document-as-unit, but the editing discipline is left to the user.

## Recommendations for crucible

Context: crucible is a typed-LLM library built on Effectful. Dynamic effects
with swappable interpreters (`Crucible.LLM` with `runLLMScripted`, cassette,
and live Anthropic/OpenAI), typed JSON boundaries via autodocodec `HasCodec`
(`Crucible.Tool`, `Crucible.Codec`), an agent loop pivoting on the pure
`Decision`/`Step` seam, and eval machinery including claim-decomposition
grounding (`Crucible.Eval.Grounding`).

What fits:

1. **A `Wiki` effect, not a wiki product.** The research says the substrate
   is small: read page, write page, list/index, search (lexical), append log.
   That is a natural crucible dynamic effect:
   `data Wiki :: Effect` with `ReadPage`, `WritePage`, `Index`, `Search`,
   `AppendLog`, plus `runWikiScripted` (pure `Map Slug Page` for tests, same
   shape as `runLLMScripted`) and `runWikiDir` (a directory of markdown
   files, compatible with Anthropic's memory tool layout and Karpathy's).
   Git, embeddings, and hosting stay out: they are application or interpreter
   concerns, and the file interpreter gets git history for free by living in
   a repo.

2. **Typed pages via `HasCodec`.** The wiki pattern's weak point is that page
   structure is enforced only by prompt. Crucible can do better at the
   boundary: a `Page meta` of slug, title, outbound links, body text, and
   typed frontmatter decoded with the existing tolerant decode path
   (`Crucible.Decode`). LangMem's "profile" (schema-validated single
   document) is precedent that typed pages are the useful unit. Entity pages,
   source pages, and synthesis pages become different `meta` types. Make
   outbound links typed, not bare slugs: the gist thread's one settled
   structural finding (pursultani, Jun 5) is that an untyped wikilink cannot
   represent a preserved contradiction or a supersession, so the link type
   (`contradicts`, `extends`, `supersedes`) belongs in the page structure,
   a small sum type in the `Page` record, while the policy for what to do
   with each type stays in the prompt.

3. **Wiki operations as `Tool`s.** Expose `read_page`, `write_page`,
   `search_pages` through the existing `tool` combinator so the stock agent
   loop can maintain a wiki. The editing policy (create at 3+ mentions,
   update on contradiction, never merge across layers, supersede with
   timestamps; the thread added mechanical variants like update-vs-create at
   60 percent entity overlap, per-page line budgets with archive compression,
   and the every-task-produces-two-outputs rule) is prompt content, the
   schema document; crucible can ship it
   as a default instruction fragment the way it ships other prompt idioms,
   but should not encode it in types. It is policy, and it will change.

4. **Grounding-gated writes.** This is the most crucible-specific
   opportunity. `Crucible.Eval.Grounding.groundingOutcome` already decomposes
   a rendered output into atomic claims and judges each against evidence.
   Apply it as a write gate: before `WritePage` commits an ingest edit,
   ground the new page text against the raw source trace; reject or flag
   edits with unsupported claims. The gist thread converged on write gating
   too, but every shipped gate is human: llm-wiki-compiler's
   `compile --review` approve/reject step, asakin's human-reviewed training
   period, watsonrm's review queue for conflicts. Automated claim-level
   verification at write time still does not exist in any surveyed system;
   the rest rely on after-the-fact lint. A typed library with eval machinery
   in-tree can make verified ingest the default rather than a manual step.

5. **Lint as `Eval` cases.** Structural lint (orphans, broken links, sparse
   pages) is pure, no LLM, and fits as plain functions over the page set.
   Semantic lint (contradiction between two pages, staleness of a claim) maps
   onto the existing judge `vote` machinery. A `lintWiki` that returns the
   existing `Report` type would reuse `runEval` directly.

What does not fit, said plainly:

- **Retrieval infrastructure.** BM25 ranking, embeddings, vector stores, rank
  fusion: out of scope. Crucible should define the `Search` operation and let
  interpreters decide; the research says titles-then-grep covers the regime
  where wikis win anyway, and the file interpreter can shell out or stay
  naive.
- **Confidence scores and decay curves** (the v2 proposals) are speculative
  and were criticized as false precision even by sympathetic readers. Skip.
  Supersession (new claim links to and marks the old one stale) is the
  defensible subset and is representable in page frontmatter.
- **Revision history as a library concept.** Git already is the provenance
  layer in every successful instance of this pattern. Modeling revisions in
  Haskell types would duplicate it badly.
- **A memory service or background consolidation scheduler** (Letta
  sleep-time agents). Crucible is a library; "run lint on a timer" is the
  caller's loop. What crucible owes the caller is that a lint pass and an
  ingest pass are ordinary `Eff` programs that run under scripted
  interpreters in tests.
- **Conversational/working memory.** The wiki pattern itself does not cover
  it (AAIF's noted gap), and crucible's `Chat`/`Agent` state already owns the
  in-session transcript. No new machinery needed.

Suggested order if any of this is picked up: typed `Page` + `Wiki` effect with
the scripted interpreter (pure, testable, small), then the directory
interpreter, then wiki-ops-as-tools, then grounding-gated writes. Each step is
useful without the next.
